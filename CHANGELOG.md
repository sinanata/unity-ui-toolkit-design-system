# Changelog

All notable changes to this project will be documented here.

This project loosely follows [Semantic Versioning](https://semver.org/) and uses the [Keep a Changelog](https://keepachangelog.com/en/1.1.0/) format.

## [1.4.7] — 2026-07-13

1.4.6 shipped a theme system with no themes in it. This ships the themes. Two come with the package — `Dark`, which is the design system's own palette, and `Light`, which is day mode — so a fresh install has something to apply instead of an empty authoring window. The showcase's twelve Codigrate palettes are baked into `ThemeData` assets too, and the showcase now paints them **through the USS cascade** rather than stamping inline styles onto every element one at a time: `:hover`, `:disabled` and `:checked` re-theme for the first time, because a real stylesheet re-resolves them for free and 1,133 lines of hand-written tree-walking never could. A `scopeSelector` field lets a theme be scoped to a class instead of `:root`, so two themes can live in one panel. Showcase grows a THEMING section that prints the live stylesheet of whatever theme you have picked. Same showcase URL, refresh to see.

It also stops theme switching from freezing the tab. Swapping themes quickly used to walk Unity's `Allocator.Domain` into its 262,144-tracked-allocation ceiling and hang, and the culprit turned out to be the cross-fade itself: a theme swap moves the same colour properties that every component animates for hover, USS cannot tell the two changes apart, and the swap quietly inherited an animation on every element at once. Swaps are instant now. Dropdowns get the same treatment they should have had all along — they open downward in screen space *and* in world space, size themselves to their list instead of a two-row floor, and follow the active theme rather than tracking only dark and light.

### Added

- **`Resources/UI/Themes/Dark` and `Resources/UI/Themes/Light`** (package). The two themes the drop-in ships with. `Dark` mirrors `DesignTokens.uss` exactly and is the asset you duplicate to start a brand theme; `Light` is scoped to `.theme-light` and is day mode. Generate or regenerate them with `Design System > Generate Built-in Themes`.
- **`ThemeData.scopeSelector`** — the USS selector the generated token block is written under. `:root` (the default) themes the whole panel. A class such as `.theme-tokyo` themes only a subtree carrying that class, which lets two themes coexist in one panel and is how a day / night PAIR works: both sheets sit on the root, and one class toggle picks which one paints. `ScopeClass` exposes the bare class name, and the applier adds and removes it for you.
- **`ThemeRuntime`** (`DesignSystem.Runtime.Theme.Applier`) — the static apply / clear seam the `ThemeApplier` components are now thin wrappers over. It exists because a host that builds its panels in code has nowhere to hang a MonoBehaviour: the showcase's world-space corridor spawns one `PanelRenderer` per section at runtime and repaints all of them from one call. **Each root remembers what it is wearing** (a `ConditionalWeakTable`, so a discarded root takes its entry with it), which means swapping, re-applying and clearing are all just `Apply` — and a caller can no longer leave a dead stylesheet behind when a `PanelRenderer` hands it a brand-new root on reload.
- **`Design System > Rebake All Themes`**, **`> Generate Built-in Themes`**, **`> Bake Showcase Themes`** — plus batch entry points (`ThemeBaker.BakeAllBatch`, `ShowcaseThemeBaker.BakeAllBatch`) so a CI run can rebuild every theme headlessly and fail the build if one does not compile.
- **THEMING showcase section.** Prints the *actual* USS the running showcase is painting with, regenerated live from the applied `ThemeData`. This works because `GenerateUssString()` is a runtime method, not an editor one — the same call the baker makes at edit time runs again in the browser, on the same asset. Pick a theme in COLORS and watch the block change.
- **12 baked Codigrate `ThemeData` assets** (`Assets/Showcase/Resources/Themes/`), generated from the bundled palette JSONs through the existing `FromCodigrate` mapping, so the baked theme and the stamped theme cannot drift.
- **57 new icons, taking the set to 120** — a gaming pack across five groups: hardware (`gamepad`, `joystick`, `dpad`, `arcade`, `cartridge`, `console`, `handheld`, `vr-headset`, `keyboard`, `mouse`, `monitor`, `headset`, `disc`), system (`power`, `battery`, `wifi`, `bug`, `save`), combat and gear (`sword` joins `axe`, `bow`, `hammer`, `pickaxe`, `swords`, `helmet`, `bomb`, `crosshair`, `skull`), fantasy and adventure (`alien`, `castle`, `crown`, `gem`, `ghost`, `mushroom`, `portal`, `potion`, `robot`, `wand`, `treasure`, `key`, `map`, `compass`), progression (`trophy`, `medal`, `leaderboard`, `star`, `coin`, `cards`, `dice`, `puzzle`) and media (`play`, `pause`, `music`, `book`, `flag`, `flag-checkered`, `rocket`, `snowflake`). All 57 arrived as SVGs with **no `.ds-icon--<name>` rule**, which made them unreachable — nothing in the system could render one. Each now has its line in `Icons.uss`'s glyph map, and the showcase's ICONS section renders all 120 rather than a hand-picked 59 (widened to 700px so they read as a grid rather than a twelve-row column; the world-space corridor harvests `.ds-section` automatically, so the gallery picks it up with no extra wiring). `docs/ICONS.md` gains a catalogue of the full set by group.
- **The WebGL build cache-busts itself.** `BuildCli` appends a per-build token to the loader, framework, data and wasm URLs in the emitted `index.html`, and exposes `window.SHOWCASE_BUILD` (also printed to the console). This is not a nicety. A WebGL build re-emits the same four filenames every time; `python -m http.server` sends no `Cache-Control` at all, which licenses a browser to cache heuristically; and Unity's loader independently keeps `.data` in IndexedDB, which a hard refresh does not clear. The failure mode is vicious rather than merely annoying: you ship a real fix, reload, still see the bug, and go hunting for a bug that is not there any more. A changing query string makes every build a distinct URL, so there is nothing left to reuse.
- **`DesignSystemEvents.DropdownPopupOpened`** (`DesignSystem.Runtime.Behaviour`) — raised once per dropdown popup instance, right after the design system has placed and sized it. Almost no host needs it: a theme that lives in a stylesheet reaches the popup by itself, because the popup is parented at panel scope and inherits any tokens installed there. It exists for the host whose palette is invented at RUNTIME, which therefore has no stylesheet to install and must paint the popup by hand — and has no other way to reach one, since Unity creates it on open and destroys it on close. Deliberately declared on a NON-generic class: a static event on `DesignSystemBehaviourBase<TComponent>` would exist once per closed generic type, so a host subscribing through the UIDocument backend would never hear an event raised through the PanelRenderer one, and the bug would present as "works on screen, does nothing in world space".

### Changed

- **The showcase paints bundled themes through the USS cascade, not inline styles.** Picking one of the twelve Codigrate palettes now adds a single pre-baked stylesheet to each panel root and lets `var()` do the rest. The inline stamper's own header explains why it existed: Unity cannot set a custom property at runtime and cannot compile a `StyleSheet` from a string in a player, so a palette that first appears at RUNTIME can only be applied by walking the tree. But these twelve never were runtime palettes — codigrate.com sends no CORS headers, so the WebGL build has *always* read them from the bundled JSON mirror. They are known at build time, so now they are baked at build time. **Randomize keeps the inline path**, because it genuinely invents its palette at runtime and there is no sheet to add. A palette with no baked asset silently falls back to inline stamping, so a bad bake degrades the showcase instead of breaking it.
- **Editor code moved out of the package's `Runtime/` folder.** `ThemeConfiguratorWindow`, `ThemeDataEditor` and `ThemePreview.uxml` now live under `Assets/DesignSystem/Editor/Theme/` alongside the `Editor/` folder the package already had, and the third assembly definition is folded into `DesignSystem.Editor`. A consumer copying `Runtime/` no longer gets a 1,170-line editor UXML they did not ask for.
- **`ThemeData` USS generation is deterministic.** The token map was a `Dictionary`, and its iteration order is an implementation detail of both the runtime and of reflection's field ordering, so the generated stylesheet came out shuffled on a different machine and the baked sub-asset churned in every diff. It is an ordered array now, and declaration order is emit order.
- **Theme Configurator** no longer demands a 1280x720 minimum window (nothing could dock it), and the ~100 lines it shared verbatim with `ThemeDataEditor` — the `[Header]` grouping, foldout state, per-type field drawing, save/revert — now live once, in `ThemeDataGUI`.

### Fixed

- **A theme swap animated, and that animation froze the tab.** Switching themes quickly flooded the console with `Allocator.Domain has reached its limit of 262144 tracked allocations` and hung the page — in world-space mode after about fifteen clicks, in screen-space after more. The cause was a cross-fade nobody had asked for. `ShowcaseTheme.uss` animated nine colour properties on `.ds-root *`, and every `ds-` component *already* animates its own colours so that hover fades. USS cannot tell the two apart: a transition fires on any change to a watched property and the cascade has no idea WHY the value moved, so a theme change — which moves those very properties, on every element, at once — silently inherited the hover animation. A running transition pins a snapshot of the element's `ComputedStyle`, and a `ComputedStyle` is a set of ref-counted native blocks taken from `Allocator.Domain`; swap faster than they retire and the live blocks stack instead of draining, straight into Unity's ceiling. **Fix**: the universal rule is deleted, and the swap turns transitions off for the frames it lands in — `ds-no-transition`, re-armed for as long as a burst of toggles continues, and removed afterwards, which cannot retro-start anything because by then the colours have settled. It takes two rules to be airtight. `transition-property: none` scores 256 and so wins its ties against the fifteen single-class component rules, but six others animate colour from a compound selector (`.ds-input .unity-text-input` scores 512, the scrollbar dragger 768) and outrank it. Rather than name every one of them and re-name them whenever the system grows another, the guard also zeroes `--transition-fast` / `--transition-medium`: a custom property inherits, so a duration of zero reaches those rules no matter how specific they are. That second rule is compounded (`.showcase-root.ds-no-transition`, 512) precisely so a `ThemeData`'s own `:root` token block — 256, and added last — cannot win the tie and hand the durations back mid-swap. Net result: a theme swap starts exactly zero transitions. Hover fades, which are a real design-system feature, are untouched.
- **The world-space corridor multiplied that cost by thirty.** `WorldSpaceCorridor` puts `ds-root` on every exhibit for its scrollbar and typography cascade, which also opted every exhibit into the universal transition — and the flat page stays loaded underneath the corridor, because `display: none` hides an element but does not exempt it from style resolution. One toggle therefore re-resolved ~2,000 elements across ~31 live panels and started up to nine transitions on each. **Fix**: as above, plus the flat page is no longer repainted while it is hidden; changes made while the corridor is up are folded into a single repaint on the way back in.
- **`.ds-drawer` carried eight colour properties in its `transition-property` purely to stay in step with that universal rule.** With the rule gone, a drawer would have been the only thing on the page still colour-fading through a swap. **Fix**: the drawer rules transition motion only (`translate`, `max-width`, `max-height`).
- **The hidden flat page was fully repainted on every theme change made in world mode.** It is `display: none`, which takes it out of layout and rendering but not out of style resolution, so its ~990 elements were re-resolved and re-transitioned invisibly on every toggle — about half the cost of a world-mode theme change, spent on pixels that do not exist. **Fix**: the corridor tells the bootstrap when it hides the page; changes made while it is hidden are folded into one repaint on the way back in.
- **The inline theme stamper pinned every root the showcase had ever built.** Its per-root registry was a strong `Dictionary<VisualElement, RootState>`, and a `PanelRenderer` hands out a brand-new root on every UI reload — thirty of them each time the world/flat toggle is pressed. Dead roots, and through each `RootState` every element they had ever touched, were never released, so their `ComputedStyle` blocks never drained either. Worse, `Revert` resolved state through a call that *minted* an entry for any root it was handed, and the cascade path calls `Revert` on every root on every paint, so panels the stamper never touches were being registered too. **Fix**: a `ConditionalWeakTable`, and `Revert` now reads without minting.
- **Re-applying the theme a root was already wearing did two full-panel restyles to produce an identical result.** `ThemeRuntime.Apply` unconditionally removed the old stylesheet and added the new one; each of those bumps the panel's style version, and a version bump re-resolves every element under it. Since a theme change repaints all thirty exhibits and most are already wearing the theme being applied, one toggle bought about sixty pointless full-panel restyles. **Fix**: `Apply` is a no-op when the root already wears that theme *and* the sheet and scope class are genuinely still on it — callers legitimately strip both, so the bookkeeping alone is not trusted.
- **Every theme change ran ~600 whole-subtree `UQuery` walks to find widgets that were not there.** The fan-out re-resolved about eighteen named elements against the flat page and each of the thirty exhibits, on every toggle. Only the COLORS exhibit has the thirteen swatch labels; the other thirty paid thirteen full tree walks each to rediscover that they have none. **Fix**: each root's theme widgets are resolved once into a weak-keyed cache. A rebuilt root is a new key, so it cannot go stale.
- **Dropdowns opened upward, two rows tall.** Both symptoms were one bug. `TuneDropdownPopup` only ever measured the space BELOW the field and clamped what it found to a 72px floor — which is two rows — so a field with little room beneath it got a sliver no matter how many choices it had; the 14-item theme picker in COLORS was the worst case. That sliver was then forced DOWNWARD anyway, past the bottom of the panel, where Unity's own `GenericDropdownMenu.EnsureVisibilityInParent` noticed it hanging outside its container and shoved it back up. We re-asserted downward on the next geometry pass, Unity moved it again, and whoever wrote last won — so the popup that this method existed to keep below the field appeared above it. **Fix**: measure both sides, size the popup to the actual item list (a 4-choice dropdown is now 4 rows, not a padded floor), cap it at 8 rows and scroll beyond that, and flip upward only when upward is genuinely roomier. Once the popup honestly fits where it is put, Unity's math agrees and stops fighting.
- **The showcase's selector-chain inspector started highlighting the entire screen.** `ds-dropdown-tuner` — the marker the runtime puts on the PANEL ROOT so the popup's hide-until-placed rule knows a tuner is present — tripped the inspector's "any class starting with `ds-` is a component" heuristic. The panel root is an ancestor of everything, so the walk-up from empty space landed on it and the inspector dutifully highlighted a component the size of the viewport. **Fix**: the inspector never treats the panel root as a component (it lives outside the document and exists to hold exactly this kind of infrastructure), and the marker classes are listed alongside the layout containers it already walks past. Both, because "`ds-*` means component" is a heuristic and the next marker somebody adds may well land on a real element.
- **Documented counts were wrong.** The docs claimed 24 components and 63 icons across `README.md`, `llms.txt`, `AGENTS.md`, `docs/ARCHITECTURE.md`, `docs/COMPONENTS.md`, `docs/ICONS.md`, the showcase itself and the Theme Configurator's preview. The real figures are **42 components and 120 icons**, and every one of those places now says so. (`CONTRIBUTING.md`'s "converted 63 source icons" is left alone: it describes what a specific past commit did, and is still true.)
- **None of the dropdown work below had ever run in a player build. Not once.** `TuneDropdownPopup` asked for the popup exactly once, on the next frame, and returned if it was not there yet — and nothing retried. That is a race, and losing it is completely silent: the tuner gives up, Unity's default placement stands, and the popup looks *fine*, because Unity's default is perfectly reasonable in screen space. The editor won that race and the player lost it, so a probe that opened dropdowns with a synthetic `SendEvent` (which attaches the popup synchronously) passed all ten of its checks while the shipped build ran none of the code under test. Every dropdown symptom reported against the WebGL showcase was one bug wearing four hats: popups flipped upward in world space (Unity clamps against a `PanelRenderer`'s 0x0 panel root), "Medium" truncated to "Me.." (Unity pins the popup to the field's width), a runtime palette never reached the popup (the event that carries it is raised by the tuner), and screen space looked correct throughout (Unity's default is good there), which is precisely what kept the real cause hidden. **Fix**: poll for the popup every frame until it attaches, capped at 30 frames. A race you lose silently is worse than a crash.
- **The popup flashed Unity's placement and Unity's palette before showing ours.** Even once the tuner runs, it cannot run soon enough: the popup does not exist when the pointer goes down, no event fires when it attaches, and Unity lays it out and *renders* it before anything of ours can reach it. So it appeared upward in the base palette and then jumped. **Fix**: `DropdownPopup.uss` keeps the popup at `opacity: 0` from the frame it is born, and `TuneDropdownPopup` adds the reveal class LAST — after placement, and after the popup-opened event has given the host its chance to paint — so the first frame anyone sees is already correct. USS is the only thing that can do this, because it is the only thing that applies on the same frame the element is created. The hide rule is gated on `ds-dropdown-tuner`, which the runtime puts on the panel root, so a project that takes the stylesheet without the behaviour gets a normal visible dropdown rather than an invisible one; `PopupProbe` asserts the reveal actually happened, because the failure mode of this trick is a dropdown that never comes back and has perfectly good geometry while it does not.
- **In world space, dropdowns kept opening upward even after all of that.** A different bug with the same symptom, and it hid behind the first one. Two things go wrong inside a `PanelRenderer`, and either alone is enough to flip a popup. First, `WorldSpaceSizeMode.Dynamic` gives the panel no viewport to fill, so `panel.visualTree` measures **0x0** while the content lays out normally as its child (measured on a live exhibit: content 240x400, exhibit root 240x460, panel root 0x0). The popup's overlay is stretched to that root, so "room below the field" came out negative for every field in the corridor, the tuner bailed out before placing anything, and Unity's default flip-up stood. Second — and this is the one worth remembering — **`worldBound` is not pixels inside a world-space panel.** The panel root carries a pixels-to-metres transform, so a laid-out 206x48 field reports a `worldBound` of about `2.1x0.4`, while `resolvedStyle` keeps reporting pixels. Any arithmetic mixing the two compares 0.4 against 42 and loses. **Fix**: the tuner does all of its geometry in the OVERLAY's local space — the space `style.top` is written in and `resolvedStyle` speaks — converting foreign rects through `WorldToLocal`, which on a screen panel is a pure translation and changes nothing. Where the panel root is degenerate it measures the room against the field's outermost laid-out ancestor instead: the exhibit's own content box, which is the visible wall panel and the only thing a world-space popup can sensibly be asked to stay inside. Verified in the corridor: all three exhibit dropdowns open DOWN, 8px under the field, inside the panel, and the 14-item theme picker renders 8 full rows.
- **The dropdown popup could not be themed, and silently ignored every palette but two.** `DropdownPopup.uss` hard-coded hex and carried its own `.theme-light` block, so it tracked dark and light and nothing else — pick Tokyo or Sequoia and the page re-themed around a popup that stayed default-dark. It was written that way because the popup is not in the document root's subtree (Unity parents it under the PANEL root, a sibling), so `DesignTokens.uss` was assumed to be out of reach. It is not: custom properties INHERIT, and the panel root is the popup's parent. **Fix**: the popup's colours are `var(--color-*)` references, and the host now installs the token block *and* the active theme's stylesheet at panel scope. A `ThemeData` compiles to a pure `{scope} { --token: value }` block, so hoisting one has none of the layout side-effects that make hoisting the full `DesignSystem.uss` a mistake. Verified end to end: with Sequoia applied, the page and the popup both resolve `--color-surface-elev` to `#394132`.
- **...except under Randomize, which is the one palette that cannot use that road.** Every other theme reaches the popup through the cascade, because a `ThemeData` is a stylesheet and the host can install it at panel scope for the popup to inherit. Randomize has no stylesheet, by construction: it invents its palette at runtime, Unity cannot set a custom property from C#, and a player build cannot compile USS from a string — which is exactly why that palette alone is still stamped inline, element by element. The popup was the one element the stamper could not reach, because Unity builds it fresh under the panel root on every open and destroys it on close: it is not in the host's tree at the moment the stamp runs, and it does not exist to be pre-styled. So the page went lime green around a popup that stayed base dark. **Fix**: the design system raises `DesignSystemEvents.DropdownPopupOpened` once per popup instance, right after placing it, and the showcase stamps that popup from the live palette — including `:hover` and `:checked`, which have to be driven by hand from there, since an inline background outranks the USS rules that would otherwise paint them. Nothing to clean up afterwards: the popup is transient. Verified in both modes: with a random palette resolving `--color-surface-elev` to `#2E2124`, the popup paints `#2E2124`.
- **`PopupProbe` had been passing the broken popup.** Its screen tests opened dropdowns on fields sitting far below the fold (the COLORS picker measured y=536 inside a 431px panel), where there is no room below by construction, and then scored the result with a `height >= 60px` check that the 72px floor cleared — plus an explicit `|| o.height <= 74f` escape hatch on the in-panel assertion. A two-row popup was green every run. **Fix**: it scrolls each field into view first, measures the real row height (28px is the `min-height`; padding and the line box resolve it to ~42px) and asserts on visible ROW COUNT against what the list, the room and the cap allow. It also covers the flip-up case, the Randomize palette, and asserts the popup wears the page's colours.
- **A dropdown popup was pinned to its field's width, so narrow dropdowns could not show their own items.** The showcase's 110px quality pickers rendered "Medium" as "Me..", and every popup in the app was one CSS-width away from the same fate. **Fix**: the popup follows the native `<select>` contract — at least as wide as the field, free to grow to fit its longest item, capped at the container's right edge. The COLORS theme picker's popup now measures 227px against a 206px field, because "Design System default" needs it.
- **A dropdown's selected value was clipped mid-glyph rather than ellipsized.** "Design System default" rendered as "Design System defa", which reads as a broken label rather than a value that does not fit. UI Toolkit does not ellipsize on its own: it needs `white-space: nowrap` *and* `overflow: hidden` *and* `text-overflow: ellipsis`, and with any one of them missing it silently falls back to clipping. **Fix**: all three, on `.ds-dropdown`'s text element. The showcase's two 110px quality pickers are also widened to 140px, which is simply what "Medium" needs.
- **The world-mode Screen/World switch never re-themed.** Pick a theme inside the corridor and everything repainted around a mode toggle that kept the palette it was born with. `SyncThemeEverywhere` paints two things — the flat page, and the corridor's exhibits — and the switch you can see in world mode is neither: the corridor hides the flat document outright, chrome and all, and the switch that replaces it belongs to `ShowcaseModeHud`, its own `UIDocument` on its own panel. **Fix**: the HUD exposes its root once built and the bootstrap paints it with the rest. Its root is force-cleared to transparent afterwards, because the Randomize path themes by stamping INLINE styles, which outrank the `ds-root--hud` rule that normally keeps the HUD from having a background — without that, choosing Randomize in world mode drops an opaque sheet of `--color-bg` across the entire gallery.
- **`PopupProbe` scored the popup's width as "same as the field", which is precisely the bug.** A popup pinned to a 110px field cannot render "Medium", and the assertion said that was correct. **Fix**: the width check now asserts the native-select contract (never narrower than the field, never past the container), and a new `textFits` check measures every item label's natural text width against the width it was actually given — so a truncated item fails the run instead of passing it. A `SCREEN narrow(quality)` step opens the exact dropdown that was broken.
- **`PopupProbe` had also never once run its world-space tests, and had talked itself into believing it could not.** They failed with `field never laid out (2.1x0.4)` and the probe's own header explained, at length, that `PanelRenderer` panels simply never resolve a layout in an editor launched into the background. That was wrong, and it was the same units bug the tuner had: `2.1x0.4` is a perfectly laid-out 206x48 field, reported in **metres**, and the probe was checking it against a pixel threshold. A false negative wrote itself into the documentation as a platform limitation, and a real world-space bug shipped green underneath it. **Fix**: the probe measures in the overlay's local space like the tuner does, and guards on `layout` rather than `worldBound`. The world tests run headlessly, and always could have.
- **The WebGL build logged `The AudioContext was not allowed to start` on a loop.** Unity's audio subsystem was enabled in a showcase that has no audio, so `_JS_Sound_Init` constructed a `WebAudio` context and armed a 400 ms `setInterval` retrying `resume()` — and Chrome refuses, and warns, on every attempt made without a user gesture. **Fix**: `Disable Unity Audio` is on. The project ships no sound and has no reason to hold an audio context open.
- **A theme's numbers came out as `0,16` on a non-English editor, and the stylesheet silently failed to parse.** `GenerateUssString` formatted floats with `{f:F0}` and `{c.a:F2}`, both of which follow the current culture. On a Turkish, German or French machine that emits `rgba(34, 197, 94, 0,16)`. This is not hypothetical for this repo; it is the author's own locale. **Fix**: every number that reaches USS now goes through one helper that uses `CultureInfo.InvariantCulture`.
- **Saving any asset in the project re-baked every theme from inside Unity's save pipeline.** An `AssetModificationProcessor.OnWillSaveAssets` hook wrote a temp `.uss`, force-imported it, added a sub-asset and deleted the file — all while a save was already in flight, which Unity does not support. It re-entered the save, and a crash mid-bake left a stray `_temp_theme.uss` sitting inside the package. **Fix**: the processor is gone. Baking happens on an explicit Save, or from `Rebake All Themes`, and the inspector now warns in plain words when a theme's values and its baked stylesheet have drifted apart.
- **The "unsaved theme" dialog fired on domain reload and on entering play mode.** `OnDisable` and `OnDestroy` do not only run when a window is closed — they run on every recompile and every Play click, so an unsaved theme turned Ctrl+R into a modal question nobody asked. **Fix**: the prompt is gated on the editor not compiling, updating or changing play mode.
- **The live preview wrote and re-imported a stylesheet six times a second, next to the theme asset, while you dragged a colour.** **Fix**: one fixed scratch path in a git-ignored folder, reused rather than deleted and recreated, and cleaned up when the window closes.
- **1.4.6's changelog credited the wrong files for the rename.** It named `CodigrateThemeApplier.cs`, `CodigrateThemeProvider.cs` and `ShowcaseDocOverlay.cs` as updated to the new namespace; none of them were touched, and none of them ever referenced the old one. `WorldSpaceCorridor.cs`, which *was* updated, went unmentioned. **Fix**: corrected, along with two other claims in that entry. The rename also left five `DesignSystemRuntime.*` references behind in USS comments and two in `docs/COMPONENTS.md`; those are swept. Historical entries keep the old names on purpose — that is what they were called at the time.

### Notes

- **Unity scores `:root` and a class selector identically.** Both come out at 256, verified by reading the specificity written into a compiled `StyleSheet`. So a theme's `:root { --token: … }` block does **not** outrank a `.theme-light { --token: … }` block — it wins only by being added later. That is why the applier adds its sheet last, and why a light-appearance palette bakes to `.theme-light` while a dark one bakes to `:root`: each override then meets an *identical* selector already in the cascade, load order alone decides, and no cross-selector specificity question is ever asked.
- **`--radius-pill-*` is deliberately not themeable.** Those are geometry constants, not brand: `--radius-pill-9` means "9px, which is half of an 18px-tall element", and Unity clamps `border-radius` per axis to half the side. A themeable pill radius is just a way to turn a pill into an ellipse. Every pill in the system is locked to an explicit px size, so changing the spacing or type scale cannot break one.
- **The dropdown popup still does not re-theme, and cannot.** `DropdownPopup.uss` uses literal hex rather than `var(--…)` on purpose, because Unity attaches the popup as a SIBLING of the panel root where the token block does not reach. Neither the inline stamper nor the cascade can colour it; it keeps its own dark and `.theme-light` chrome. Unchanged in this release, and called out because it is the one thing a reader would reasonably expect the cascade to have fixed.

## [1.4.6] — 2026-07-13

The design system gains a first-class theme-authoring pipeline: a `ThemeData` asset stores every token, a split-pane `ThemeConfiguratorWindow` previews changes live, and a `ThemeApplier` component swaps themes at runtime *through* the USS cascade — it adds one baked stylesheet, and every `var(--…)` reference in the system re-resolves against it. The monolithic `DesignSystemRuntime` is restructured into `DesignSystemBehaviour` with a cleaner namespace hierarchy and directory layout. Showcase assets are moved out of the shippable package folder — the package now contains only what a consumer copies.

### Added

- **`ThemeData` asset** (`Create > Design System > Theme`). ### Added

- **Meter** (`Controls.uss`) — `.ds-meter`, the game bar: health, mana, XP, stamina, shield. `.ds-progress` is an 8 px app-chrome bar that hides its title outright, so every game bar got hand-rolled, and the obvious hand-roll is broken: a `position: absolute` Label inside a 12 px `overflow: hidden` track is taller than the track that clips it, so the bar ships with the top two pixels of some text peeking out of it. `.ds-meter` is sized around its label instead, and the label is a plain flow child, so it can't be clipped by construction.     - **The label is a sibling of the fill, and comes after it.** UI Toolkit has no `z-index` — later in the DOM is the only way to paint over the fill — and a Label placed *inside* the fill is clipped to the fill's width and slides around as the value changes.

Same fix pattern as `.ds-chip > .unity-label` (1.1.0): Unity's `<Label>` ships a 1 px default margin and measures a line box taller than its glyphs, and inside a 20 px `overflow: hidden` track that clips the very text the meter exists to show.
- **Wrap control** (`Typography.uss`) — `.ds-nowrap` (`white-space: nowrap; flex-shrink: 0`; one line, refuses to be shrunk) and `.ds-truncate` (`nowrap` + `overflow: hidden` + `text-overflow: ellipsis` at `flex-shrink: 1`; one line, takes the width it's given, ellipsizes the rest). Every prose class sets `white-space: normal` (1.2.0), which is right for a paragraph and wrong for a slot name, a stat value, or a menu entry. Reach for `.ds-nowrap` on any label sitting in a row beside a fixed-size element, and `.ds-truncate` where the text is user data of unknown length and the column width is what matters. See Fixed for what the missing half was doing.
- **`.ds-info-row__suffix`** (`Cards.uss`) — trailing slot after the value, for deltas, units, and buffs. Reserves its own 36 px column so rows with and without one still agree on where the value ends. Hide the content, never the slot: `opacity: 0` keeps the layout (a fade-in mustn't reflow the row), where `display: none` collapses the column and shoves that one row's value right.

### Fixed

- **A side nav could not switch anything.** `.ds-side-nav` is a tab strip — vertical tabs — but `EnsureTabs` only ever looked for `.ds-tabs`, so a Settings screen with a Graphics / Audio / Controls rail rendered the Graphics pane and left Audio and Controls permanently unreachable. Nothing in the markup was wrong; there was no code path. **Fix**: `EnsureSideNav` (see Added) takes a `.ds-tabpanels` sibling and maps the Nth `.ds-nav-item` to the Nth `.ds-tabpanel` through the same switcher as `.ds-tabs`. A side nav with no panel container is left alone, exactly like a filter-chip tab strip.
- **Drag ghost inflated to a full-screen rectangle whenever the dragged item was sized in percent.** `.ds-drag-ghost` mirrors the source item's classes — that's what makes a chip's ghost look like that chip (1.4.3) — and it's a trap for anything sized relative to its container. An inventory item is typically `width: 100%; height: 100%` of its 72 px cell; the ghost is reparented to the `.ds-root`, where the same rule resolves to 100 % of the screen. **Fix**: `MakeGhost` reads `resolvedStyle.width` / `.height` off the element the pointer went down on (already laid out, so the values are real rather than NaN) and pins the ghost to that pixel size inline, which outranks every class rule it just copied.
- **Dropdown popup chrome shipped as a showcase file, so every other project got an unstyled popup.** `.ds-dropdown` outside this repo opened a Unity-grey menu with chunky default scrollbars, and no markup a screen author could write would fix it: Unity parents the popup as a SIBLING of `.ds-root` (`GenericDropdownMenu`, see 1.2.0), where no `<Style>`-imported sheet can reach it. **Fix**: the popup chrome is a package file now — `DesignSystem/Resources/UI/Styles/DesignSystem/DropdownPopup.uss`, was `Showcase/Resources/ShowcaseDropdownPopup.uss` — and the host-side attach is documented in its header and in `docs/COMPONENTS.md`. - **The animal-detail hero poked out of its card's rounded top corners in the walkable gallery** (since 1.4.3, when world space shipped). `.ds-animal-detail` is a rounded `overflow: hidden` card and `.ds-animal-detail__hero` is a square-cornered band flush against its top edge, which is fine anywhere the clip honours the radius and wrong in a world-space panel, where it doesn't (see Notes). Low contrast (`--color-surface-elev` on `--color-surface`) is the only reason it went unreported. **Fix**: the hero rounds its own top corners, same as `.ds-meter__fill` and `.ds-progress`'s fill.

### Notes

- **A world-space `PanelRenderer` panel clips `overflow: hidden` to the rectangle and ignores the `border-radius`.** Screen space honours it, so a square-cornered child flush against a rounded parent looks perfect on the flat page and hangs out of the curve in the 3D gallery. This is what the first cut of `.ds-meter` did: its fill leaned on the track's `overflow: hidden` to become a pill, and in the gallery its square corners poked out of the track's half-circle left cap and read as the bar overflowing to the left. Rectangular clips are unaffected (the drawers rely on one) — it is only the rounding that is dropped. **The rule: a child that reaches a ROUNDED parent's edge carries the rounding itself.** `.ds-progress` already did; `.ds-meter__fill` and `.ds-animal-detail__hero` do now.
- **`DropdownField` pre-selects with `index`, not `value`.** `DropdownField.value` is not a UXML attribute, so Unity ignores it without a word and the field renders blank. `<ui:DropdownField choices="Low,Medium,High" index="2" class="ds-dropdown"/>`.
- **The runtime font atlas still has no em dash** (1.4.3 hit this in the hero texts). The new showcase sections reintroduced them and they render as blank gaps; showcase copy is ASCII-only again.

### Notes

- **Why not override `--color-*` at runtime?** Unity 6's public UI Toolkit API has no path for setting CSS custom properties on a `VisualElement` — `style` is typed and covers only the built-in properties; you can only get values via `RegisterCallback<CustomStyleResolvedEvent>`. The internal `StyleVariableContext` is reflection-only and not stable across versions. An explicit per-class walk-and-stamp is therefore the only public-API path that hits every visible element while keeping the DS USS as the single source of truth for everything else (spacing, radii, transitions, layout). If a future Unity release exposes `style.SetCustomProperty(name, value)`, the applier collapses to a single root-level call.
- **Static codigrate mirror exists for a reason.** The upstream JSON is unauthenticated and Cloudflare-fronted, but the host doesn't return `Access-Control-Allow-Origin` headers, so browser-context fetches (WebGL) deterministically fail the CORS preflight. The bundled `Resources/CodigrateThemes/` mirror is the WebGL path of record; Editor and Standalone go live so upstream edits land without a rebuild.
- **Hover swaps re-installed via pointer callbacks.** Inline styles beat the `:hover` USS rule that would otherwise drive hover swaps on brand buttons, so `Apply` registers `PointerEnter` / `PointerLeave` callbacks per brand-variant button that stamp the hover colour on enter and the base colour on leave. `Revert` unregisters them so a theme swap doesn't leak duplicate listeners.

Bundling the rules into an existing imported file sidesteps the cache cliff entirely. The matrix is `direction × mode`:

The showcase title `Unity 6 UI Toolkit Design System` clipped at the "m" of "System" instead of wrapping. Root cause: UI Toolkit's `<Label>` defaults to `white-space: nowrap`, and the design-system typography classes never overrode it. On a 5K Mac everything fit on one line so the bug stayed invisible; on a phone any heading > one viewport-width-of-text overflowed silently. **Fix**: `Typography.uss` now sets `white-space: normal` on every prose class (`.ds-h1` / `.ds-h2` / `.ds-h3` / `.ds-body-1` / `.ds-body-2` / `.ds-caption` / `.ds-section-label`). Single-word labels (button text, badge / tag / chip captions) use their own component classes and aren't affected.
- **Dropdown popup rendered Unity's stock light theme regardless of the day / night toggle.** `BasePopupField` opens a `GenericDropdownMenu` whose container / item / label / checkmark classes (`.unity-base-dropdown__*`) had no design-system rules, so Unity's default white-on-light theme leaked through — visibly wrong against the dark surface, and impossible to fix from the toggle because the popup never received the `theme-light` cascade either. **Fix**: new rules in `Inputs.uss` paint the popup background and border from `--color-surface-elev` / `--color-border`, items from `--color-text-primary`, hover from `--color-surface-hover`, and the selected row from `--color-primary-soft`. `WireThemeToggle` in `ShowcaseBootstrap` now also toggles the `theme-light` class on `root.panel.visualTree` — Unity attaches the popup as a SIBLING of `.ds-root` under `panel.visualTree`, so a class scoped to root alone never reaches it.
- **Dropdown chevron and selected-item checkmark were soft on HiDPI.** Both come from Unity's default theme as 1× raster sprites; the `panelSettings.scale = DPR` multiplier then bilinearly upscales them, fuzzing every Retina / DPR-3 display. **Fix**: `Inputs.uss` now points `.ds-dropdown .unity-base-popup-field__arrow` at `Textures/Icons/chevron_down`, and the new `Showcase/Resources/ShowcaseDropdownPopup.uss` (see below) points `.unity-base-dropdown__checkmark` at `Textures/Icons/check` — both with `-unity-background-scale-mode: scale-to-fit` so the Vector Graphics package rasterises them at the actual rendered pixel count.
- **Dropdown popup rendered Unity's white default theme regardless of theme + got two further regressions while iterating to a working version.** Verified against `UnityCsReference/Modules/UIElements/Core/Controls/GenericDropdownMenu.cs`. Three separate gotchas:
  1. **Class names.** Unity uses `unity-base-dropdown__checkmark` (NOT `__item-checkmark`) for the glyph, and `:checked` pseudo-state on the item (NOT a `--checked` modifier class) for the selected state.
  2. **Stylesheet scope.** The popup is added under `m_TargetElement.GetRootVisualContainer()` — a SIBLING of `.ds-root`, not a descendant — so any rules placed in the design-system bundle (loaded via the UXML `<Style>` import, scoped to `rootVisualElement` only) never matched the popup. Pulling the entire `DesignSystem.uss` + `ShowcaseTheme.uss` bundle to panel-root scope as a fix made things worse: one of the bundled rules over-claimed the popup's layout. The dedicated `Showcase/Resources/ShowcaseDropdownPopup.uss` ships only the `unity-base-dropdown__*` rules with literal hex colours and a `.theme-light` block, so panel-root scope contains exactly what the popup needs and nothing else.
  3. **The outer `.unity-base-dropdown` element is sized to fill the entire panel by Unity, intentionally** — `GenericDropdownMenu.DropDown` sets its `style.width / height` to `panel.layout.width / height` so it acts as a transparent click-capture overlay that dismisses the popup on outside-click. Painting `background-color` on it covered the entire viewport. The actual visible popup chrome (background, border, radius, padding) belongs on `.unity-base-dropdown__container-outer`, which Unity positions at the source field's worldBound. Moved the chrome rules accordingly; the outer overlay now stays transparent.

  **Fix**: corrected class names + new dedicated `Showcase/Resources/ShowcaseDropdownPopup.uss` loaded at `panel.visualTree` scope by `ShowcaseBootstrap`, with chrome on `__container-outer` and the outer `.unity-base-dropdown` left transparent. `WireThemeToggle` adds the `theme-light` class to BOTH `.ds-root` AND `panel.visualTree` so the popup picks up the override.
- **Slider and MinMaxSlider tracks went invisible; thumbs were white-on-white in light theme.** Two pre-existing bugs surfaced by today's testing (likely always present; the dark default theme hid the thumb issue by accident). The scrollbar rules in `Controls.uss` (added in 1.1.0) used `.ds-root .unity-base-slider__tracker` and `.ds-root .unity-base-slider__dragger`, both of which match ANY slider inside `.ds-root` — including `.ds-slider` and the inner sliders of `.ds-range`. Same specificity as the per-component rules, declared LATER, so the scrollbar's `background-color: transparent` overrode the regular slider's surface-elev track colour. The `.ds-slider .unity-base-slider__dragger` and `.ds-range .unity-min-max-slider__min-thumb / __max-thumb` rules also hard-coded `#FFFFFF`, invisible against the light-theme `--color-surface` background. **Fix**: scrollbar rules now require `.ds-root .unity-scroll-view__vertical-scroller`/`__horizontal-scroller` as the parent, so they only match sliders that are children of a `ScrollView`'s scroller. Hard-coded `#FFFFFF` thumbs swapped to `var(--color-text-primary)`, which inverts cleanly between themes (off-white on dark, dark on light).

### Known caveats / what isn't here yet

- **Gamepad detection requires a user gesture.** Browsers gate the Gamepad API behind one keystroke or click on the page (anti-fingerprinting). Plug in a controller, click the canvas once, then D-pad / sticks register. Not a build issue; documented browser behaviour.
- **Initial DPR is captured once.** `panelSettings.scale` is set at bootstrap; if the user drags the browser window between mixed-DPI monitors mid-session the panel won't re-scale until reload. Add a periodic DPR re-read if anyone reports it.
- **Tab order traverses every demo button.** The BUTTONS section ships ~50 visual-state mock buttons (Hover / Pressed / Disabled demos for each variant). They're real `<ui:Button>` elements, so gamepad / Tab walks through every one. We left them focusable on purpose — it's the cleanest way to inspect how the focus ring renders against each variant — but if the navigation feels too long, mark `.ds-btn--demo-hover` / `.ds-btn--pressed` / `[enabled="false"]` as `focusable=false` from the bootstrap.
