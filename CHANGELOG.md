# Changelog

All notable changes to this project will be documented here.

This project loosely follows [Semantic Versioning](https://semver.org/) and uses the [Keep a Changelog](https://keepachangelog.com/en/1.1.0/) format.

## [1.4.6] — 2026-07-13

The design system gains a first-class theme-authoring pipeline: a `ThemeData` asset stores every token, a split-pane `ThemeConfiguratorWindow` previews changes live, and a `ThemeApplier` component swaps themes at runtime without touching the USS cascade. The monolithic `DesignSystemRuntime` is restructured into `DesignSystemBehaviour` with a cleaner namespace hierarchy and directory layout. Showcase assets are moved out of the shippable package folder — the package now contains only what a consumer copies.

### Added

- **`ThemeData` asset** (`Create > Design System > Theme`). A `ScriptableObject` that stores every design token — brand/semantic colors, text colors, surface tiers, rarity colors, typography sizes, radii, spacing, border widths, and motion durations — as inspector-editable fields. Each field maps to a `var(--...)` USS custom property via a static `FieldToUss` dictionary. `GenerateUssString()` emits the full palette as a `:root { ... }` CSS custom property block. Companion `.uss` files are embedded next to the asset via `AssetDatabase.CreateAsset` during save.
- **`ThemeConfiguratorWindow`** (`Design System > Theme Configurator`). A split-pane editor window: the right pane is an `IMGUIContainer` showing the `ThemeData` inspector grouped by `[Header]` categories; the left pane holds a live UXML preview (`ThemePreview.uxml`) that re-generates and hot-reloads the theme stylesheet on every token change (debounced at 150 ms via `schedule.Execute + EditorApplication.delayCall`). Double-click any `ThemeData` asset in the Project window to open it here (`OnOpenAsset` callback). A toolbar `ObjectField` accepts a custom `.uxml` asset for the preview panel (GUID persisted to `EditorPrefs`); the bundled `ThemePreview.uxml` inside the `Editor/` folder is the fallback. Save/Revert buttons, dirty-state tracking, and an unsaved-changes dialog on close complete the authoring loop.
- **`ThemePreview.uxml`** (`Assets/DesignSystem/Runtime/Theme/Data/Editor/ThemePreview.uxml`). A 1170-line simplified showcase snapshot — color swatches, typography scale samples, buttons of every variant, inputs, toasts, badges, tags, chips, a spinner, and a skeleton — designed to give immediate visual feedback while tuning a theme. Loads `DesignSystem.uss` via asset-database path so token references resolve in the editor.
- **`ThemeDataEditor`** (`[CustomEditor(typeof(ThemeData))]`). Groups serialized properties by `[Header]` attribute into foldable categories (Brand & Semantic, Text, Surfaces, Rarity, Typography, Radius, Spacing, etc.). Save button generates and embeds the companion `.uss`; Revert restores the last-saved JSON snapshot. "Open in Theme Configurator" button launches the split-pane window. Unsaved-changes dialog on inspector deselect.
- **`ThemeApplierBase<TComponent>`** and two concrete implementations. Attach to a `UIDocument` or `PanelRenderer` and the component adds/removes the theme's companion `.uss` stylesheet from the root `VisualElement` at runtime (adds when theme is set, removes the previous sheet on swap). Version-guarded for `PanelRenderer` via `RegisterUIReloadCallback` so theme persists across UI reloads; `UIDocument` variant polls briefly when the visual tree hasn't materialised yet.
    - `DesignSystem.Runtime.Theme.Applier.UIDocument.ThemeApplier` — `[AddComponentMenu("Design System/UI Document Theme Applier")]`
    - `DesignSystem.Runtime.Theme.Applier.PanelRenderer.ThemeApplier` — `[AddComponentMenu("Design System/Panel Renderer Theme Applier")]`, `#if UNITY_6000_5_OR_NEWER`

### Changed

- **`DesignSystemRuntime` renamed and restructured to `DesignSystemBehaviour`.** Directory layout moved from flat `Runtime/DesignSystemRuntime.cs` to `Runtime/Behaviour/` → `DesignSystemBehaviourBase.cs`, `UIDocument/DesignSystemBehaviour.cs`, `PanelRenderer/DesignSystemBehaviour.cs`.
- **`DesignSystemShowcase.uxml` moved** from `Assets/DesignSystem/Resources/UI/Styles/DesignSystem/` to `Assets/Showcase/Resources/`. The showcase scene, scripts, world-space corridor, and demo assets are part of the host project — not the shippable package a consumer copies. The UXML stylesheet reference was updated from a relative `project://database/` path to the full asset-database path with GUID and fileID so it resolves correctly from its new location.
- **Showcase scripts** (`ShowcaseBootstrap.cs`, `CodigrateThemeApplier.cs`, `CodigrateThemeProvider.cs`, `ShowcaseDocOverlay.cs`) updated to reference the new `DesignSystem.Runtime.Behaviour.UIDocument` namespace instead of the old `DesignSystem.Runtime` namespace.

## [1.4.5] — 2026-07-13

`.ds-meter` lands: the health / mana / XP bar that `.ds-progress` was never meant to be. `.ds-nowrap` and `.ds-truncate` add wrap control for the short fixed labels that make up most of a UI, and `.ds-info-row__suffix` adds the trailing column that keeps a stat list's numbers aligned. The side nav becomes a real tab strip — it switches panels through the same runtime switcher as `.ds-tabs`, which it had never been wired to — the drag ghost stops inflating to full-screen, and the dropdown popup chrome moves out of the showcase and into the package, so a consuming project gets a styled popup instead of a Unity-grey one. Showcase grows VERTICAL TABS (SIDE NAV) and GAME METERS sections in both the flat page and the walkable gallery. Same showcase URL, refresh to see.

### Added

- **Meter** (`Controls.uss`) — `.ds-meter`, the game bar: health, mana, XP, stamina, shield. `.ds-progress` is an 8 px app-chrome bar that hides its title outright, so every game bar got hand-rolled, and the obvious hand-roll is broken: a `position: absolute` Label inside a 12 px `overflow: hidden` track is taller than the track that clips it, so the bar ships with the top two pixels of some text peeking out of it. `.ds-meter` is sized around its label instead, and the label is a plain flow child, so it can't be clipped by construction. Sizes are `--sm` (10 px — a status pip, too short to carry a label), default (20 px — the HUD bar), and `--lg` (24 px — a boss bar or a full-width cast bar). Fill colour by role: `.ds-meter--secondary / --tertiary / --warning / --danger`. `.ds-meter__fill.is-warning` / `.is-danger` are live states to flip from game code as a value crosses a threshold; they're declared after the role modifiers, so `is-danger` wins on a `--secondary` meter.
    - **The label is a sibling of the fill, and comes after it.** UI Toolkit has no `z-index` — later in the DOM is the only way to paint over the fill — and a Label placed *inside* the fill is clipped to the fill's width and slides around as the value changes.
    - `.ds-meter__label` zeroes its margin / padding and re-applies `-unity-text-align: middle-center`. Same fix pattern as `.ds-chip > .unity-label` (1.1.0): Unity's `<Label>` ships a 1 px default margin and measures a line box taller than its glyphs, and inside a 20 px `overflow: hidden` track that clips the very text the meter exists to show.
- **Wrap control** (`Typography.uss`) — `.ds-nowrap` (`white-space: nowrap; flex-shrink: 0`; one line, refuses to be shrunk) and `.ds-truncate` (`nowrap` + `overflow: hidden` + `text-overflow: ellipsis` at `flex-shrink: 1`; one line, takes the width it's given, ellipsizes the rest). Every prose class sets `white-space: normal` (1.2.0), which is right for a paragraph and wrong for a slot name, a stat value, or a menu entry. Reach for `.ds-nowrap` on any label sitting in a row beside a fixed-size element, and `.ds-truncate` where the text is user data of unknown length and the column width is what matters. See Fixed for what the missing half was doing.
- **`.ds-info-row__suffix`** (`Cards.uss`) — trailing slot after the value, for deltas, units, and buffs. Reserves its own 36 px column so rows with and without one still agree on where the value ends. Hide the content, never the slot: `opacity: 0` keeps the layout (a fade-in mustn't reflow the row), where `display: none` collapses the column and shoves that one row's value right.
- **`DesignSystemRuntime.EnsureSideNav(root)`** (package runtime, both backends) — wires every `.ds-side-nav` to its `.ds-tabpanels` sibling. Auto-attached from `EnsureAll` (1.4.4), idempotent via a marker class, re-scanned on the 250 ms tick. See Fixed.

### Changed

- **`.ds-info-row__label` and `.ds-info-row__value` are `white-space: nowrap; flex-shrink: 0`** — an info row is a label and a value, never a paragraph. `.ds-info-row__value` also gains `-unity-text-align: middle-right` and a 40 px `min-width`, so a stack of rows agrees on where the numbers end whether or not any given row carries a suffix.
- **`WireTabs` and `WireSideNav` are one switcher.** Both delegate to a single private `WirePanelSwitcher(strip, itemClass)`: same sibling `.ds-tabpanels` lookup, same direct-children panel list, same Nth-item-shows-Nth-panel activation. The item query is untyped because a `.ds-tab` is a `Button` while a `.ds-nav-item` is usually a plain `VisualElement` — buttons wire through `clicked`, everything else through `ClickEvent`.
- **Showcase**: new **VERTICAL TABS (SIDE NAV)** and **GAME METERS (HEALTH / MANA / XP)** sections, in the flat page and in the walkable gallery.

### Fixed

- **A side nav could not switch anything.** `.ds-side-nav` is a tab strip — vertical tabs — but `EnsureTabs` only ever looked for `.ds-tabs`, so a Settings screen with a Graphics / Audio / Controls rail rendered the Graphics pane and left Audio and Controls permanently unreachable. Nothing in the markup was wrong; there was no code path. **Fix**: `EnsureSideNav` (see Added) takes a `.ds-tabpanels` sibling and maps the Nth `.ds-nav-item` to the Nth `.ds-tabpanel` through the same switcher as `.ds-tabs`. A side nav with no panel container is left alone, exactly like a filter-chip tab strip.
- **Drag ghost inflated to a full-screen rectangle whenever the dragged item was sized in percent.** `.ds-drag-ghost` mirrors the source item's classes — that's what makes a chip's ghost look like that chip (1.4.3) — and it's a trap for anything sized relative to its container. An inventory item is typically `width: 100%; height: 100%` of its 72 px cell; the ghost is reparented to the `.ds-root`, where the same rule resolves to 100 % of the screen. **Fix**: `MakeGhost` reads `resolvedStyle.width` / `.height` off the element the pointer went down on (already laid out, so the values are real rather than NaN) and pins the ghost to that pixel size inline, which outranks every class rule it just copied.
- **Dropdown popup chrome shipped as a showcase file, so every other project got an unstyled popup.** `.ds-dropdown` outside this repo opened a Unity-grey menu with chunky default scrollbars, and no markup a screen author could write would fix it: Unity parents the popup as a SIBLING of `.ds-root` (`GenericDropdownMenu`, see 1.2.0), where no `<Style>`-imported sheet can reach it. **Fix**: the popup chrome is a package file now — `DesignSystem/Resources/UI/Styles/DesignSystem/DropdownPopup.uss`, was `Showcase/Resources/ShowcaseDropdownPopup.uss` — and the host-side attach is documented in its header and in `docs/COMPONENTS.md`. Deliberately not `@import`ed into `DesignSystem.uss`: importing it there would attach it at `.ds-root` scope, where it still wouldn't match.

- **The animal-detail hero poked out of its card's rounded top corners in the walkable gallery** (since 1.4.3, when world space shipped). `.ds-animal-detail` is a rounded `overflow: hidden` card and `.ds-animal-detail__hero` is a square-cornered band flush against its top edge, which is fine anywhere the clip honours the radius and wrong in a world-space panel, where it doesn't (see Notes). Low contrast (`--color-surface-elev` on `--color-surface`) is the only reason it went unreported. **Fix**: the hero rounds its own top corners, same as `.ds-meter__fill` and `.ds-progress`'s fill.

### Notes

- **A world-space `PanelRenderer` panel clips `overflow: hidden` to the rectangle and ignores the `border-radius`.** Screen space honours it, so a square-cornered child flush against a rounded parent looks perfect on the flat page and hangs out of the curve in the 3D gallery. This is what the first cut of `.ds-meter` did: its fill leaned on the track's `overflow: hidden` to become a pill, and in the gallery its square corners poked out of the track's half-circle left cap and read as the bar overflowing to the left. Rectangular clips are unaffected (the drawers rely on one) — it is only the rounding that is dropped. **The rule: a child that reaches a ROUNDED parent's edge carries the rounding itself.** `.ds-progress` already did; `.ds-meter__fill` and `.ds-animal-detail__hero` do now.
- **`DropdownField` pre-selects with `index`, not `value`.** `DropdownField.value` is not a UXML attribute, so Unity ignores it without a word and the field renders blank. `<ui:DropdownField choices="Low,Medium,High" index="2" class="ds-dropdown"/>`.
- **The runtime font atlas still has no em dash** (1.4.3 hit this in the hero texts). The new showcase sections reintroduced them and they render as blank gaps; showcase copy is ASCII-only again.

## [1.4.4] — 2026-07-13

Tab strips switch content. `.ds-tabs` styled the selected state and stopped there, so a tabbed screen shipped as four buttons above one long scroll of every section stacked together; add a `.ds-tabpanels` sibling and the Nth `.ds-tab` now shows the Nth `.ds-tabpanel`, wired by the runtime. `.ds-root--hud` gives a HUD a transparent screen root, so UI over gameplay stops rendering as a dark slab. Both are live in both showcase modes — the flat page grows a TABBED PANELS section, and the same section hangs in the walkable gallery as a panel you can click through. Same showcase URL, refresh to see.

### Added

- **Tab panels** (`TabsAndFilters.uss`) — `.ds-tabpanels` / `.ds-tabpanel`. Add a `.ds-tabpanels` container as a sibling of the `.ds-tabs` strip (the canonical layout puts it directly after) and the Nth `.ds-tab` shows the Nth `.ds-tabpanel`. Positional, not id-based: nothing to name, nothing to target, nothing to keep in sync. `is-active` remains the only state, so a C# controller or a pure-USS author can drive the same markup by flipping the class.
    - A `.ds-tabs` with **no** `.ds-tabpanels` sibling is left alone — a filter-chip strip over a single list is still a valid use, and those drive their own logic off `is-active`.
- **`DesignSystemRuntime.EnsureTabs(root)`** (package runtime, both backends) — wires the clicks. Auto-attached by `InitFor`, idempotent via a marker class, and re-scanned on the existing 250 ms tick so lazily-cloned screens are covered.
- **`DesignSystemRuntime.EnsureScrollAutoHide(root)`** (package runtime, both backends) — see Fixed.
- **`DesignSystemRuntime.EnsureAll(root)`** (package runtime) — runs every auto-attached behaviour in one call. The seam a host that drives its own `PanelRenderer` reload callback is meant to call instead of hand-picking helpers; see Fixed.
- **`.ds-root--hud`** (`DesignTokens.uss`) — a transparent, content-sized screen root (`background-color: rgba(0, 0, 0, 0)`, `flex-grow: 0`). `.ds-root` is the scope the rest of the system hangs off — the text ramp, plus the scrollbar, focus-ring, and transition families, which all select through it — but it also paints `--color-bg` edge to edge. That is right for a full-screen menu and wrong for a health bar: a HUD built on plain `ds-root` renders as a dark slab over the game. Compose `class="ds-root ds-root--hud"` for anything the game world must show through.

### Changed

- **Showcase**: new **TABBED PANELS** section in the flat page and in the walkable gallery. The world-space HUD (the mode switch, the hints bar, the touch sticks) is built on `ds-root--hud` now, which is what lets its mode switch be a real `.ds-tabs` instead of a hand-styled lookalike.

### Fixed

- **Auto-hiding scrollbars only auto-hid for mouse users.** `WireScrollAutoHide` adds the `is-scrolling` marker that the touch half of the `.ds-scroll--auto-hide` rule needs (1.3.0), and `InitFor` never called it — the helper had been dead since it was written, so the modifier worked via `:hover` and nowhere else. **Fix**: `EnsureScrollAutoHide(root)` (see Added), auto-attached alongside the other behaviours.
- **New runtime behaviours reached screen space but not world space.** A host that drives its own `PanelRenderer` reload callback (the showcase corridor) can't use the runtime MonoBehaviour, so it called the `Ensure*` helpers by hand — and that hand-written list quietly stopped being the whole system every time a behaviour was added. Tab panels and scroll auto-hide shipped dead in the 3D gallery for exactly this reason. **Fix**: `EnsureAll(root)` (see Added) is the single seam every host calls, so a new behaviour lands in every host at once.
- **A tab strip could seize an unrelated view's panels.** The panel container was resolved with a subtree search from the strip's parent, so a filter-chip strip sharing a container with a tabbed view further down the page — the showcase's own Screen / World mode switch sits in the same scroll view as every demo section — could bind to that view's panels and then fight its real strip over which one is active. **Fix**: the container lookup is strictly the strip's sibling and the panel list is strictly that container's direct children, so a tabbed view nested inside a tab panel works.

## [1.4.3] — 2026-07-06

The showcase gains a second mode: a walkable 3D gallery where all 29 sections hang as live, fully interactive world-space panels (`PanelRenderer`, Unity 6000.5+) on the walls of a lit corridor. Dropdown popups now behave like web selects in both modes, drag & drop works everywhere, and the repo ships metadata for contributors and AI coding assistants. First release published to OpenUPM. Same showcase URL, refresh to see.

### Added

- **World-space showcase gallery** (Unity 6000.5+ only; the flat page is unchanged on older versions). A segmented `Screen Space / World Space` switch under the hero swaps the flat page for a first-person corridor where every section is its own world-space panel (one `PanelRenderer` + world-space `PanelSettings` per section, `WorldSpaceSizeMode.Dynamic`). Everything stays interactive: buttons, inputs, dropdowns, drawers, drag & drop, and the theme system (day/night, Codigrate palettes, random) work from the flat page or from inside the gallery, mirrored both ways.
    - **Movement**: WASD or arrows to walk, hold right-mouse or Q/E to look, Esc or the Screen Space tab exits. Typing into an exhibit's text field never moves the camera. Touch devices get two pointer-captured on-screen sticks (walk + look) so camera input and component taps cannot conflict; portrait screens widen the FOV.
    - **Exhibit sizing** comes from the flat page's resolved layout, because UXML-authored inline styles are not visible to the C# `style` API (see Notes in the repo memory of 1.4.2 era; `resolvedStyle` on the live document is the only truthful source).
    - Panel roots carry `ds-root` so the scoped styling families (slim scrollbars, 14px baseline, transitions, focus ring) match the flat page, and each world panel loads the dropdown-popup stylesheet at its own panel scope.
- **`DesignSystemRuntime.EnsureDropdownMenus(root)`** (package runtime, both backends). Wires every `.ds-dropdown` `DropdownField` so its popup follows the web select model: always opens DOWNWARD from the field, pinned to the field's width, capped to the space below it (up to 320px), scrolling vertically only, re-asserted after Unity's own positioning pass (`GenericDropdownMenu` otherwise sizes the popup to the raw option list and flips or slides it when that doesn't fit). Auto-wired by `InitFor` on both backends and by the showcase's world exhibits; idempotent via a marker class.
- **URP render pipeline assets** in `Assets/Settings/` (`URPRenderPipelineAsset` + Forward renderer), now actually referenced by `GraphicsSettings` and both quality levels. See Fixed for what the previous dangling references did.
- **`Assets/Editor/PopupProbe.cs`** editor diagnostic: enters play mode, opens every kind of showcase dropdown (flat page and world exhibits), and asserts the popup geometry with numbers (downward, snug gap, height cap, field-width pin, no horizontal scroller, popup inside the panel). Complements `PipelineProbe` (which render pipeline actually resolves) and `WorldLayoutProbe` (flat vs world layout numbers).
- **Repo meta for contributors and AI assistants**: `AGENTS.md` and `llms.txt` teach Copilot, Cursor, Codex, and Claude Code the `ds-` classes, tokens, and runtime helpers; `.github/` ships bug-report and feature-request issue forms, an issue config, a PR template, and `copilot-instructions.md`; `CITATION.cff` and `SECURITY.md` cover citation metadata and a disclosure policy; README gains a badge row, a table of contents, and an AI-assistant pointer.

### Changed

- **Hero promo buttons.** Steam, App Store, and Google Play buttons (the game is live on all three stores) replace the old wishlist button.
- **Showcase scene ships skybox-free**: solid-color camera background and flat ambient; corridor surfaces clone a committed URP Simple Lit material (`CorridorLit.mat`, emission keyword pinned) kept alive by a tiny scene renderer; the corridor shell is segmented so Forward per-object light limits illuminate the full hall.
- **Showcase GIF** re-recorded from the current build (720px, 8fps, 112-color palette, 9.4MB, inside GitHub's 10MB render limit) and README restructured for the UPM package split (generic runtime base with UIDocument and PanelRenderer backends) and the world-space gallery.
- **`package.json`**: fixed the misspelled `keyword` field to `keywords` and expanded the list (`ui-toolkit`, `unity6`, `design-system`, `design-tokens`, `component-library`, `dark-theme`, and friends) for UPM and GitHub discoverability. Version bumped to 1.4.3 for the OpenUPM release.

### Fixed

- **Dropdown popups opened upward with Unity's default chunky scrollbars on both axes.** Tall option lists (the Colors theme picker) flipped upward pinned to the panel edge on the flat page, and inside world panels, which are only as tall as their own section, every popup got clamped to whatever sliver of panel remained. Fixed by `EnsureDropdownMenus` (see Added) plus popup-stylesheet work: the popup attaches outside `.ds-root`, so the design system's slim-scrollbar restyle never reached it. `ShowcaseDropdownPopup.uss` now mirrors the slim 8px pill scrollbar for both themes, removes the horizontal scroller outright, and ellipsizes options longer than the field.
- **Drag ghost rendered as a huge unstyled band; first frame placed it at NaN.** The ghost was parented to `panel.visualTree`, which sits ABOVE the element where every design-system stylesheet attaches, so no rule ever matched it (including its `position: absolute`): it entered layout flow and stretched the panel's full width, and the initial centering math divided by a not-yet-resolved `resolvedStyle.width`. The ghost now spawns under the nearest `.ds-root` ancestor, mirrors the dragged item's classes and children (a chip's ghost looks like that chip, with a primary lift border), centers via `translate(-50%,-50%)`, dims the source item while dragging (new `.is-dragging` state class), and cleans up on any pointer-capture loss.
- **The project silently ran the built-in render pipeline.** `GraphicsSettings` and both quality levels referenced URP asset GUIDs that existed nowhere on disk, so every URP-tagged shader compiled to zero variants and runtime-instantiated materials rendered magenta in WebGL builds (the corridor's walls). Real URP pipeline + Forward renderer assets now live in `Assets/Settings/` and are wired up.
- **Repeated Screen/World mode toggling crashed WebGL with an uncatchable `RuntimeError: divide by zero`.** A rebuilt world panel resolves a zero size for one frame and the engine's integer division on it is a wasm-spec trap that `wasmArithmeticExceptions` cannot suppress. World panels are never SetActive-toggled anymore: the 3D geometry toggles normally while panels switch `style.visibility`.
- **Applying a Codigrate or random palette across the gallery reverted every panel except the last.** `CodigrateThemeApplier` kept one global stamp registry and `Apply` began with a global revert. Bookkeeping is per root now; day/night, Codigrate, and random themes work from either mode with the cloned theme controls mirrored both ways.
- **Missing-glyph tofu in hero texts.** The runtime font atlas lacks the em dash, minus sign, and ellipsis glyphs; replaced with ASCII-safe text. The mode-band caption no longer wraps to two lines.

### Notes

- **Never put `Universal Render Pipeline/Lit` in Always Included Shaders on Unity 6000.5.2f1**: the WebGL build segfaults deterministically while enumerating its variant space. With a real pipeline asset and a scene-referenced material it is also unnecessary.
- **Script-launched background editors cannot verify world-space layout.** Screen-space UIDocuments lay out fine without rendering, but `PanelRenderer` panels stay 2x0 stubs unless a Game view genuinely renders, even when the Game view is acquired, focused, and `Repaint()`-pumped every tick. `PopupProbe`'s screen assertions carry the numeric load; world mode is verified in the WebGL build.

## [1.4.2] — 2026-07-05

### Added

- **`Assets/Showcase/Resources/DefaultPanelSettings.asset`** — pre-configured `PanelSettings` asset for the showcase. `ShowcaseBootstrap.MakePanelSettings` now tries `Resources.Load<PanelSettings>("DefaultPanelSettings")` and falls back to `ScriptableObject.CreateInstance<PanelSettings>()` if the asset is missing. This has been done because there must be at least one PanelSettings asset in project, otherwise ICU Data becames not available when advanced text option enabled in the Editor and on the builds Unity throws errors constantly and cuts the UI process.
- **Editor menu: `Assets/Design System/Add Stylesheet To Selected Layout Asset`** — right-click any `.uxml` file in the Project window and the helper injects `<Style src="DesignSystem.uss">` inside the `<ui:UXML>` element. Validates that the selection is a UXML file, resolves `DesignSystem.uss` by GUID, checks for duplicate references (idempotent), regexes the style tag just after the opening `<ui:UXML ...>` line, then writes the updated template back to disk. Disabled (greyed out) when no UXML asset is selected. Lives in `EditorHelpers.cs`.
- **Unity Package Manager integration** — `Assets/DesignSystem/package.json` (`com.sinanata.designsystem`) declares version, description, and Unity minimum version so the design system can be consumed as an immutable package via Git URL. `README.md` updated with Option C instructions: add `"com.sinanata.designsystem": "https://github.com/sinanata/unity-ui-toolkit-design-system.git?path=/Assets/DesignSystem"` to `Packages/manifest.json`, or use the Package Manager window's "Add package from git URL..." dialog. Ideal for projects that prefer package-manager workflows and don't need to edit the system source.
- **`[assembly: AlwaysLinkAssembly]`** in `Assets/DesignSystem/Runtime/AssemblyInfo.cs` — prevents Unity's managed-linker from stripping the `DesignSystem.Runtime` assembly in IL2CPP builds. Without it, `[RuntimeInitializeOnLoadMethod]` methods in the runtime can be stripped and the auto-attach registration never fires.
- **`DesignSystem.Runtime.asmdef`** — assembly definition file giving the runtime folder its own assembly, isolating it from the rest of the project and making it referenceable by other asmdefs.
- **`DesignSystemRuntimeBase<TComponent>`** — abstract base class replacing the single monolithic `DesignSystemRuntime`. Moves spinner rotation, toggle-knob injection, skeleton shimmer, drawer wiring, scroll auto-hide, and drag & drop wiring into one shared base. Two concrete child classes cover both Unity UI Toolkit backends:
    - **`DesignSystem.Runtime.UIDocumentRuntime.DesignSystemRuntime`** — auto-attaches to every `UIDocument` in the scene via `[RuntimeInitializeOnLoadMethod]`.
    - **`DesignSystem.Runtime.PanelRendererRuntime.DesignSystemRuntime`** — for `PanelRenderer`-backed screens (Unity 6000.5+). Listens to `RegisterUIReloadCallback` so toggles and spinners re-inject after a UI reload; version-gated so repeated callbacks don't double-initialize. Guarded behind `#if UNITY_6000_5_OR_NEWER`.
- **Drag & drop** — `.ds-draggable` / `.ds-drop-zone` / `.is-drag-over` / `.ds-drag-ghost` (in `Controls.uss`) plus auto-wiring in `DesignSystemRuntime.EnsureDraggables`: mark items `.ds-draggable` and containers `.ds-drop-zone` and the runtime handles pointer drag, a floating `.ds-drag-ghost`, drop-zone highlighting, and reparent-on-drop — no code. Game inventories with custom split/merge/transfer logic drive their own pointer handling and reuse only the `.ds-drag-ghost` / `.is-drag-over` visuals. Rendered in the showcase's DRAG & DROP section.
- **`.ds-tooltip`** — a floating info-surface component (in `Overlays.uss`). A positioned card (the **consumer** sets the screen position, e.g. mouse-follow with edge-flip in C#) on the elevated surface tier with a *strong* border so it reads clearly when floating over busy content rather than nested inside a card. Slots: `__title` (body-1 bold), `__subtitle` (caption), `__body` (body-2, wraps), `__divider` (hairline), and `__row` (`__row-label` left + `__row-value` right) for stat lines. ### Notes

- **Why not override `--color-*` at runtime?** Unity 6's public UI Toolkit API has no path for setting CSS custom properties on a `VisualElement` — `style` is typed and covers only the built-in properties; you can only get values via `RegisterCallback<CustomStyleResolvedEvent>`. The internal `StyleVariableContext` is reflection-only and not stable across versions. An explicit per-class walk-and-stamp is therefore the only public-API path that hits every visible element while keeping the DS USS as the single source of truth for everything else (spacing, radii, transitions, layout). If a future Unity release exposes `style.SetCustomProperty(name, value)`, the applier collapses to a single root-level call.
- **Static codigrate mirror exists for a reason.** The upstream JSON is unauthenticated and Cloudflare-fronted, but the host doesn't return `Access-Control-Allow-Origin` headers, so browser-context fetches (WebGL) deterministically fail the CORS preflight. The bundled `Resources/CodigrateThemes/` mirror is the WebGL path of record; Editor and Standalone go live so upstream edits land without a rebuild.
- **Hover swaps re-installed via pointer callbacks.** Inline styles beat the `:hover` USS rule that would otherwise drive hover swaps on brand buttons, so `Apply` registers `PointerEnter` / `PointerLeave` callbacks per brand-variant button that stamp the hover colour on enter and the base colour on leave. `Revert` unregisters them so a theme swap doesn't leak duplicate listeners.
- **Dropdown popup under codigrate.** Unity adds `GenericDropdownMenu` as a SIBLING of `rootVisualElement` under `panel.visualTree`, so the applier's `root.Query(...)` doesn't reach it — the popup chrome stays at the existing `ShowcaseDropdownPopup.uss` defaults regardless of which palette is active. A second pass keyed to `panel.visualTree` would close the gap; deferred.

### Added

- **Drawer rules** appended to `Controls.uss` (rather than living in their own file). Bundling the rules into an existing imported file sidesteps the cache cliff entirely. The matrix is `direction × mode`:
    - **Direction** modifiers: `.ds-drawer--top`, `.ds-drawer--right`, `.ds-drawer--bottom`, `.ds-drawer--left`. Each pins the drawer to its named edge and chooses the off-edge resting state (`translate: 0 -100%`, `100% 0`, `0 100%`, `-100% 0` respectively).

    - **State** is toggled by either an `is-open` class on a `.ds-drawer-wrap` ancestor (recommended; one class flip animates drawer + backdrop together) or by `ds-drawer--open` on the drawer itself (freestanding usage). Both selectors are wired in the USS so authors can pick whichever fits their tree.

The runtime helper toggles `pickingMode` in lockstep with the open class because UI Toolkit's `opacity: 0` does NOT disable pointer picking — an invisible-but-rendered backdrop would otherwise shadow the burger button beneath it.

### Fixed

- **Components rendered at half size on Retina Macs and 1/3 size on iPhones.** The WebGL template sets `canvas.width = window.innerWidth × window.devicePixelRatio` so Unity renders into a HiDPI buffer for crisp text, which means `Screen.width` reports the *buffer* pixel count (5120 on a 5K iMac with default scaling, 1290 on an iPhone 14 Pro Max), not CSS pixels. With `PanelScaleMode.ConstantPixelSize` and the default `panelSettings.scale = 1`, every `36-px` `.ds-btn` was rendering at 36 buffer-px = 18 CSS-px on DPR-2 displays and 12 CSS-px on DPR-3 phones. **Fix**: `MakePanelSettings` now sets `ps.scale = GetEffectiveDpr()`. On WebGL the DPR comes from the new `WebGLDevicePixelRatio.jslib` bridge; on Standalone (where the bridge returns nothing) it derives from `Screen.dpi / 96` and floors at 1 so non-HiDPI desktops stay unchanged. Same `36 panel-px = 36 CSS-px` rendering on every device, with the HiDPI buffer preserved for sharpness.
- **Mobile layout never triggered on iPhone 14 Pro Max** (and other DPR-3 devices). `ApplyMobileClass` was comparing `Screen.width < 768` — 1290 buffer-px is never < 768, so the `.mobile` class wasn't applied even though the CSS viewport is 430 px wide. **Fix**: read `rootVisualElement.layout.width` instead, which is in panel coordinates = CSS pixels on every device (because `ps.scale = DPR`). Falls back to `Screen.width / DPR` only if layout hasn't resolved yet.
- **Doc-overlay panel docked off-screen on every Retina display** once the panel-scale fix landed. `ShowcaseDocOverlay.PositionPanelNear` was clamping with `if (left + PANEL_WIDTH > Screen.width)` — that mixes panel-coordinate `left` (CSS-px) with `Screen.width` (buffer-px), pushing the panel ~`(DPR - 1) × screen-width` past the right edge on Retina. **Fix**: read `_overlayDoc.rootVisualElement.layout.width / .height` for clamping; both `worldBound` and `Screen` are now in the same unit space.
- **Stale `Screen.width` semantics comment** in the WebGL template's `<style>` block. Read "Unity's `Screen.width` reads the CSS size" — true with `matchWebGLToCanvasSize: false`, false with the default `true`. Reworded to call out the buffer-vs-CSS distinction so a contributor doesn't reach for `Screen.width < 768` again.
- **Headings and captions ran off the right edge of the viewport on phones** — reported on Samsung S25 Ultra (412 CSS-px portrait), reproducible on any narrow device. The showcase title `Unity 6 UI Toolkit Design System` clipped at the "m" of "System" instead of wrapping. Root cause: UI Toolkit's `<Label>` defaults to `white-space: nowrap`, and the design-system typography classes never overrode it. On a 5K Mac everything fit on one line so the bug stayed invisible; on a phone any heading > one viewport-width-of-text overflowed silently. **Fix**: `Typography.uss` now sets `white-space: normal` on every prose class (`.ds-h1` / `.ds-h2` / `.ds-h3` / `.ds-body-1` / `.ds-body-2` / `.ds-caption` / `.ds-section-label`). Single-word labels (button text, badge / tag / chip captions) use their own component classes and aren't affected.
- **Dropdown popup rendered Unity's stock light theme regardless of the day / night toggle.** `BasePopupField` opens a `GenericDropdownMenu` whose container / item / label / checkmark classes (`.unity-base-dropdown__*`) had no design-system rules, so Unity's default white-on-light theme leaked through — visibly wrong against the dark surface, and impossible to fix from the toggle because the popup never received the `theme-light` cascade either. **Fix**: new rules in `Inputs.uss` paint the popup background and border from `--color-surface-elev` / `--color-border`, items from `--color-text-primary`, hover from `--color-surface-hover`, and the selected row from `--color-primary-soft`. `WireThemeToggle` in `ShowcaseBootstrap` now also toggles the `theme-light` class on `root.panel.visualTree` — Unity attaches the popup as a SIBLING of `.ds-root` under `panel.visualTree`, so a class scoped to root alone never reaches it.
- **Dropdown chevron and selected-item checkmark were soft on HiDPI.** Both come from Unity's default theme as 1× raster sprites; the `panelSettings.scale = DPR` multiplier then bilinearly upscales them, fuzzing every Retina / DPR-3 display. **Fix**: `Inputs.uss` now points `.ds-dropdown .unity-base-popup-field__arrow` at `Textures/Icons/chevron_down`, and the new `Showcase/Resources/ShowcaseDropdownPopup.uss` (see below) points `.unity-base-dropdown__checkmark` at `Textures/Icons/check` — both with `-unity-background-scale-mode: scale-to-fit` so the Vector Graphics package rasterises them at the actual rendered pixel count.
- **Dropdown popup rendered Unity's white default theme regardless of theme + got two further regressions while iterating to a working version.** Verified against `UnityCsReference/Modules/UIElements/Core/Controls/GenericDropdownMenu.cs`. Three separate gotchas:
  1. **Class names.** Unity uses `unity-base-dropdown__checkmark` (NOT `__item-checkmark`) for the glyph, and `:checked` pseudo-state on the item (NOT a `--checked` modifier class) for the selected state.
  2. **Stylesheet scope.** The popup is added under `m_TargetElement.GetRootVisualContainer()` — a SIBLING of `.ds-root`, not a descendant — so any rules placed in the design-system bundle (loaded via the UXML `<Style>` import, scoped to `rootVisualElement` only) never matched the popup. Pulling the entire `DesignSystem.uss` + `ShowcaseTheme.uss` bundle to panel-root scope as a fix made things worse: one of the bundled rules over-claimed the popup's layout. The dedicated `Showcase/Resources/ShowcaseDropdownPopup.uss` ships only the `unity-base-dropdown__*` rules with literal hex colours and a `.theme-light` block, so panel-root scope contains exactly what the popup needs and nothing else.
  3. **The outer `.unity-base-dropdown` element is sized to fill the entire panel by Unity, intentionally** — `GenericDropdownMenu.DropDown` sets its `style.width / height` to `panel.layout.width / height` so it acts as a transparent click-capture overlay that dismisses the popup on outside-click. Painting `background-color` on it covered the entire viewport. The actual visible popup chrome (background, border, radius, padding) belongs on `.unity-base-dropdown__container-outer`, which Unity positions at the source field's worldBound. Moved the chrome rules accordingly; the outer overlay now stays transparent.

  **Fix**: corrected class names + new dedicated `Showcase/Resources/ShowcaseDropdownPopup.uss` loaded at `panel.visualTree` scope by `ShowcaseBootstrap`, with chrome on `__container-outer` and the outer `.unity-base-dropdown` left transparent. `WireThemeToggle` adds the `theme-light` class to BOTH `.ds-root` AND `panel.visualTree` so the popup picks up the override.
- **Slider and MinMaxSlider tracks went invisible; thumbs were white-on-white in light theme.** Two pre-existing bugs surfaced by today's testing (likely always present; the dark default theme hid the thumb issue by accident). The scrollbar rules in `Controls.uss` (added in 1.1.0) used `.ds-root .unity-base-slider__tracker` and `.ds-root .unity-base-slider__dragger`, both of which match ANY slider inside `.ds-root` — including `.ds-slider` and the inner sliders of `.ds-range`. Same specificity as the per-component rules, declared LATER, so the scrollbar's `background-color: transparent` overrode the regular slider's surface-elev track colour. The `.ds-slider .unity-base-slider__dragger` and `.ds-range .unity-min-max-slider__min-thumb / __max-thumb` rules also hard-coded `#FFFFFF`, invisible against the light-theme `--color-surface` background. **Fix**: scrollbar rules now require `.ds-root .unity-scroll-view__vertical-scroller`/`__horizontal-scroller` as the parent, so they only match sliders that are children of a `ScrollView`'s scroller. Hard-coded `#FFFFFF` thumbs swapped to `var(--color-text-primary)`, which inverts cleanly between themes (off-white on dark, dark on light).

### Known caveats / what isn't here yet

- **Gamepad detection requires a user gesture.** Browsers gate the Gamepad API behind one keystroke or click on the page (anti-fingerprinting). Plug in a controller, click the canvas once, then D-pad / sticks register. Not a build issue; documented browser behaviour.
- **Initial DPR is captured once.** `panelSettings.scale` is set at bootstrap; if the user drags the browser window between mixed-DPI monitors mid-session the panel won't re-scale until reload. Add a periodic DPR re-read if anyone reports it.
- **Tab order traverses every demo button.** The BUTTONS section ships ~50 visual-state mock buttons (Hover / Pressed / Disabled demos for each variant). They're real `<ui:Button>` elements, so gamepad / Tab walks through every one. We left them focusable on purpose — it's the cleanest way to inspect how the focus ring renders against each variant — but if the navigation feels too long, mark `.ds-btn--demo-hover` / `.ds-btn--pressed` / `[enabled="false"]` as `focusable=false` from the bootstrap.
